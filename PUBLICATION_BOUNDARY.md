# PUBLIC 發佈邊界

本工作區只承載「意脈系統」可公開使用的產品版本內容。它與 OPS 工作區分離，不是 OPS 的鏡像、備份或交接出口。

## 核心原則

PUBLIC 的主體是意脈系統：公開產品說明、可重現示例、未來 package / CLI、技能封裝與使用者文件。

Agent Handoff Kit 只可作為本倉庫的 AI 協作、交接與治理底層，不可在公開文件中被描述成產品本體。

這代表：

- 可以公開：產品用途、使用方式、可重現示例、公開文件、清理後的技能或 package 內容。
- 可以提及但不可喧賓奪主：底層 AI 協作機制、健康檢查、跨 session 接續。
- 不可公開：OPS 的交接歷史、session log、內部研發輸出、私人知識庫資料、本機私人路徑、憑證或未清理素材。

## OPS 與 PUBLIC 的關係

| 工作區 | 功能 | 發佈角色 |
|---|---|---|
| OPS | 私有研發、驗收、內部治理、證據沉澱 | 產生候選成果，不直接公開 |
| PUBLIC | 公眾 GitHub / npm 產品發佈面 | 承接清理後內容，服務公眾理解、安裝與使用意脈系統 |

同步只允許由 OPS 到 PUBLIC 的受控單向流程。PUBLIC 不自動同步 OPS，也不作為 OPS 的備份或鏡像。

## 從 OPS 到 PUBLIC 的同步流程

當 OPS 開發完成並通過驗收後，PUBLIC 發佈前必須完成：

1. 在 OPS 列明候選成果與驗收證據。
2. 將候選成果分類為可公開、需改寫、不可公開。
3. 移除 OPS 交接紀錄、研發證據、私人知識庫資料、本機私人路徑、憑證和未授權素材。
4. 把可公開內容轉化為意脈系統的產品材料，例如 README、公開文件、示例、技能封裝或 npm package 檔案。
5. 在 PUBLIC 執行健康檢查、私隱掃描、差異檢查與必要產品測試。
6. 檢查通過後才可提交 PUBLIC。
7. push、tag、release、package publish 或部署必須另行取得明確授權。

## 可公開

- 產品用途、安裝方式、升級方式、操作流程和限制。
- 已清理的公開示例。
- 不依賴私人知識庫或本機路徑的通用文件。
- 經確認可公開授權的程式碼、技能檔和模板。
- 必要時可簡短說明底層 AI 協作機制，但不得把它寫成產品主體。

## 不可公開

- OPS 治理檔、交接紀錄、session log、內部研發輸出。
- 私人知識庫識別碼、本機絕對路徑、個人資料、未清理截圖。
- 憑證、token、API key、OAuth secret、refresh token 或任何可還原機密。
- 未確認授權的第三方全文、圖片、PDF 或資料集。

## 發佈說明規則

PUBLIC 每次撰寫 release notes，必須面向非開發者用戶，而不是寫技術改動流水帳。固定格式如下：

| 欄位 | 要回答的問題 |
|---|---|
| 這次更新帶來什麼 | 用一句話說明用戶看到的變化。 |
| 對用戶有什麼直接好處 | 說明節省了什麼、降低了什麼風險、或多了什麼能力。 |
| 用戶需要做什麼 | 明確寫「不需要行動」或列出下一步。 |
| 安裝與升級 | 說明新用戶如何安裝意脈系統，已有用戶如何安全升級。 |
| 注意事項 | 寫清楚限制、邊界或仍未支援的能力。 |

不要把 release notes 寫成 commit 清單、內部任務列表、程式碼差異或治理流水帳。

## 發佈前檢查

每次公開發佈前至少確認：

- 沒有 OPS 私人資料被搬入。
- 沒有 credential 或可還原機密。
- README 沒有宣稱未完成能力。
- 安裝與升級說明以意脈系統自身 package、CLI 或正式 agent 指令為入口；若底層依賴 Agent Handoff Kit，只能作依賴說明。
- 示例可由公眾理解，不需要 Adam 私人環境。
- GitHub repo、tag、release、package publish 另行取得明確授權。
