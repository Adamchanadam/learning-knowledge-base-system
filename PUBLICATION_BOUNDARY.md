# 公開發佈邊界

本工作區只承載「個人知識與品味脈絡系統」可公開使用的產品版本內容。它與 營運工作區 工作區分離，不是 營運工作區 的鏡像、備份或交接出口。

## 核心原則

公開倉庫的主體是個人知識與品味脈絡系統：公開產品說明、可重現示例、未來 套件 / 命令列入口、技能封裝與使用者文件。

Agent Handoff Kit 只可作為本倉庫的人工智能協作、交接與治理底層，不可在公開文件中被描述成產品本體。

這代表：

- 可公開內容：產品用途、使用方式、可重現示例、公開文件、清理後的技能或套件內容。
- 可以提及但不可喧賓奪主：底層人工智能協作機制、健康檢查、跨 對話 接續。
- 不可公開內容：營運工作區的交接歷史、交接日誌、內部研發輸出、私人知識庫資料、本機私人路徑、憑證或未清理素材。

## 營運工作區與公開工作區的關係

| 工作區 | 功能 | 發佈角色 |
|---|---|---|
| 營運工作區 | 私有研發、驗收、內部治理、證據沉澱 | 產生候選成果，不直接公開 |
| 公開工作區 | 公眾 GitHub / npm 產品發佈面 | 承接清理後內容，服務公眾理解、安裝與使用個人知識與品味脈絡系統 |

同步只允許由營運工作區到公開工作區的受控單向流程。公開工作區不自動同步營運工作區，也不作為 營運工作區 的備份或鏡像。

## 從 營運工作區到公開工作區 的同步流程

當 營運工作區開發完成並通過驗收後，公開發佈前必須完成：

1. 在營運工作區列明候選成果與驗收證據。
2. 將候選成果分類為可公開、需改寫、不可公開。
3. 移除 營運工作區交接紀錄、研發證據、私人知識庫資料、本機私人路徑、憑證和未授權素材。
4. 把可公開內容轉化為個人知識與品味脈絡系統的產品材料，例如 README、公開文件、示例、技能封裝或 npm 套件 檔案。
5. 在 公開工作區執行健康檢查、私隱掃描、差異檢查與必要產品測試。
6. 檢查通過後才可提交公開工作區。
7. 推送、標籤、發佈頁、套件發佈 或部署必須另行取得明確授權。

## 可公開

- 產品用途、安裝方式、升級方式、操作流程和限制。
- 已清理的公開示例。
- 不依賴私人知識庫或本機路徑的通用文件。
- 經確認可公開授權的程式碼、技能檔和模板。
- 必要時可簡短說明底層人工智能協作機制，但不得把它寫成產品主體。

## 不可公開

- 營運工作區治理檔、交接紀錄、交接日誌、內部研發輸出。
- 私人知識庫識別碼、本機絕對路徑、個人資料、未清理截圖。
- 憑證、權杖、應用程式介面金鑰、OAuth 密鑰、更新權杖 或任何可還原機密。
- 未確認授權的第三方全文、圖片、PDF 或資料集。

## 發佈說明規則

公開工作區每次撰寫 發佈說明，必須面向非開發者用戶，而不是寫技術改動流水帳。固定格式如下：

| 欄位 | 要回答的問題 |
|---|---|
| 這次更新帶來什麼 | 用一句話說明用戶看到的變化。 |
| 對用戶有什麼直接好處 | 說明節省了什麼、降低了什麼風險、或多了什麼能力。 |
| 用戶需要做什麼 | 明確寫「不需要行動」或列出使用者需要做的事。 |
| 安裝與升級 | 說明新用戶如何安裝個人知識與品味脈絡系統，已有用戶如何安全升級。 |
| 注意事項 | 寫清楚限制、邊界或仍未支援的能力。 |

不要把 發佈說明 寫成 commit 清單、內部任務列表、程式碼差異或治理流水帳。

## 發佈前檢查

每次公開發佈前至少確認：

- 沒有 營運工作區私人資料被搬入。
- 沒有 憑證 或可還原機密。
- README 沒有宣稱未完成能力。
- 安裝與升級說明以個人知識與品味脈絡系統自身 套件、命令列入口 或正式 代理指令為入口；若底層依賴 Agent Handoff Kit，只能作依賴說明。
- 示例可由公眾理解，不需要 Adam 私人環境。
- GitHub 倉庫、標籤、發佈頁、套件發佈 另行取得明確授權。
