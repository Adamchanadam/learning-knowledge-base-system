# 公開文案語氣規格

本規格用於「個人知識與品味脈絡系統」的公開 README、說明、教學、示例介紹、發佈說明和未來安裝文件。它的作用是把目前 README 已確認合適的語氣固定下來，避免日後公開文件又變回內部規格、技術流水帳或人工智能拼貼式文字。

## 核心語氣

公開文案要讓一般讀者一眼明白：這是一個幫人把看到的內容快速內化，變成自己能用、有啟發性知識的工具。

語氣要淺白、穩定、有人味。句子可以短，但要說清楚產品幫人完成什麼，不要用抽象詞堆疊出氣氛。文案要像向一個認真閱讀、但不熟悉本專案的人解釋，而不是向開發者交代任務。

## 標準說明段

需要用一段文字介紹產品時，優先使用以下語義，可按版面長短微調，但不得改成技術安裝說明或治理說明：

> 這是一個幫你把你看到的內容，通過 AI 幫你快速內化，變成自己能用、有啟發性的知識的工具。
>
> 你給它一篇文章、一個 URL 或一段文字，它不單只是幫你寫摘要，而是會保存可追溯的來源，整理重點，拆出知識點，生成幫你快速看懂原文的資訊圖，再判斷這內容跟你的工作、學習、生活選擇或審美取向有什麼關係。
>
> 時間久了，它會慢慢形成一套屬於你的知識脈絡和品味脈絡，讓你不是被平台推薦牽著走，而是更清楚自己為什麼採納、深讀、甚至拒絕某些內容。

## 必須保留的產品重點

公開文案描述產品時，必須保留以下重點。篇幅不夠時可合併，但不可整批消失：

- 不是只做摘要，而是幫使用者內化內容。
- 會保存可追溯來源。
- 會整理重點、拆出知識點。
- 會生成幫人快速看懂原文的資訊圖。
- 會判斷內容和使用者的工作、學習、生活選擇或審美取向有什麼關係。
- 長期會形成知識脈絡和品味脈絡。
- 目標是幫使用者更清楚採納、深讀、暫存或拒絕內容，而不是被平台推薦牽著走。

## 資訊圖的寫法

資訊圖是核心交付，不是附加裝飾。公開文案提到系統能力時，不能只寫摘要、知識點或 Obsidian 入庫，而漏掉資訊圖。

正確寫法：

- 「生成幫你快速看懂原文的資訊圖。」
- 「用資訊圖把原文裡難吸收的概念、流程、關係、差異或重點變成一眼能理解的視覺整理。」
- 「資訊圖服務人理解原文，不取代原文真源，也不混入個人推論。」

避免寫法：

- 「附帶生成圖片。」
- 「產出視覺素材。」
- 「用圖表美化內容。」

## 用詞規則

- 優先用「你」「內容」「來源」「知識點」「資訊圖」「知識脈絡」「品味脈絡」「採納」「深讀」「拒絕」這類讀者能直接理解的詞。
- 可以保留必要專有名詞，例如 AI、URL、Obsidian、Markdown、Bases、npm、Agent Handoff Kit。
- 不把 `dev/`、交接、治理、doctor、tag、release、push、internal log 等內部詞寫入一般公開說明。
- 不用「本工具賦能」「智能化閉環」「多模態知識中台」「一站式解決方案」這類空泛語。
- 不用一串名詞堆出句子；每句要能回答「它幫我做什麼」。

## 結構規則

公開 README、教學或說明文應按以下順序組織：

1. 先用淺白段落說明它是什麼。
2. 再說它處理一篇文章、一個 URL 或一段文字時會做什麼。
3. 必須提到資訊圖，而且要說明它幫人快速理解原文。
4. 再說長期會形成知識脈絡和品味脈絡。
5. 最後才說目前公開狀態、尚未提供的安裝或套件能力。

不要一開始就講底層協作機制、治理檔、安裝依賴或倉庫內部結構。

## 禁止事項

- 不把 Agent Handoff Kit 寫成產品本體。
- 不把 README 寫成安裝底層協作工具的教學。
- 不把公開文案寫成開發任務清單、發佈操作清單或內部驗收流水帳。
- 不宣稱尚未完成的能力，例如完整安裝包、可直接安裝的命令列入口、可重現公開示例或 npm 套件。
- 不把資訊圖說成 AI 的資料來源；資訊圖是給人看的理解交付。
- 不把所有內容都硬說成品味分析；事實資料、工作流程和學習材料要先忠實整理。

## 驗收問題

任何公開文案完成前，至少回答以下問題：

1. 一般讀者是否不用懂內部技術，也能明白這工具幫他做什麼？
2. 文案是否清楚說明它不是只做摘要？
3. 文案是否有提到資訊圖，並說明資訊圖的真正用途？
4. 文案是否說清楚長期知識脈絡和品味脈絡的價值？
5. 文案是否避免了底層協作工具喧賓奪主？
6. 文案是否沒有宣稱未完成能力？
